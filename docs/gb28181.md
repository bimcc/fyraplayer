# FyraPlayer GB28181 网关适配方案

> Updated: 2026-05-17
> Scope: FyraPlayer does not implement a browser-side GB28181 protocol stack. It integrates with a server-side GB28181 gateway and plays the standard media URL returned by that gateway.

## 1. 边界结论

GB28181 的主体能力属于服务端，不属于浏览器播放器端。

服务端网关负责：

- SIP 注册、鉴权、目录、INVITE/BYE、心跳。
- RTP/PS 收流、丢包处理、时间戳修正。
- PS 解复用、音视频转码或转封装。
- 输出浏览器可消费的标准媒体协议：FLV、MPEG-TS、HLS、WebRTC 或 fMP4。
- 设备状态、PTZ、录像回放、级联等平台逻辑。

FyraPlayer 负责：

- 调用后端 `invite` 控制接口。
- 从后端响应中提取可播放媒体 URL。
- 播放后端已经转出的 FLV/TS URL。
- 暴露 `gb:bye`、`gb:ptz`、`gb:query`、`gb:keepalive` 控制入口。
- 将控制面错误和会话字段整理成播放器事件或异常。

不在播放器端实现：

- SIP/RTP/PS 协议栈。
- GB28181 PS 直解。
- 浏览器端自定义 GB 帧协议。
- G.711/PS/H.265 等国标媒体兼容兜底逻辑。

## 2. 当前实现

`tech-gb28181.ts` 是网关适配器，不是完整媒体 Tech。

`load()` 行为：

1. 调用 `source.control.invite`。
2. 按 `responseMapping` 或默认字段解析媒体 URL、`callId`、`ssrc`、`streamId`。
3. 将后端返回的 FLV/TS URL 交给 `mpegts.js`/MSE 播放。
4. 记录会话字段，供后续控制接口使用。

控制入口：

- `player.control('gb:invite', payload)`
- `player.control('gb:bye', payload?)`
- `player.control('gb:ptz', payload)`
- `player.control('gb:query', payload?)`
- `player.control('gb:keepalive', payload?)`

`gb:bye` 和 `gb:ptz` 默认会带上 `deviceId`、`channelId`、`callId`、`ssrc`、`streamId`。

## 3. Source 形状

```ts
type Gb28181Source = {
  type: 'gb28181';
  url: string; // optional fallback media URL when invite is absent or response has no URL
  control: {
    invite: string;
    bye: string;
    ptz?: string;
    query?: string;
    keepalive?: string;
  };
  controlRequest?: {
    headers?: Record<string, string>;
    credentials?: RequestCredentials;
  };
  gb: {
    deviceId: string;
    channelId: string;
    ssrc?: string;
    streamMode?: 'UDP' | 'TCP-Active' | 'TCP-Passive';
    transport?: 'udp' | 'tcp';
    expires?: number;
  };
  responseMapping?: {
    url?: string;
    callId?: string;
    ssrc?: string;
    streamInfo?: string;
    streamId?: string;
  };
  format?: 'flv' | 'ts';
  preferTech?: 'gb28181';
};
```

`format` 只描述网关返回的浏览器可播放封装：

- `flv` -> `mpegts.js` FLV MSE path。
- `ts` -> `mpegts.js` MPEG-TS MSE path。

## 4. 后端响应契约

推荐响应：

```json
{
  "stream_id": "34020000001320000001",
  "callId": "call-123",
  "ssrc": "001122",
  "play_urls": {
    "urls": {
      "ws_flv": "wss://media.example.com/live/34020000001320000001.flv",
      "ws_ts": "wss://media.example.com/live/34020000001320000001.ts"
    }
  }
}
```

默认 URL 解析顺序：

1. `responseMapping.url`
2. `url`
3. `wsUrl`
4. `play_urls.urls.ws_flv`
5. `play_urls.ws_flv`
6. `play_urls.urls.flv`
7. `play_urls.urls.ws_ts`
8. `play_urls.ws_ts`
9. `source.url`

认证可以通过 `source.controlRequest.headers` 或 `source.controlRequest.credentials` 提供。

## 5. 验证口径

自动化测试只验证播放器适配契约：

- invite 请求体和鉴权配置。
- URL 字段映射和默认回退。
- FLV/TS MSE 播放分派。
- PTZ/BYE/query/keepalive 控制调用。
- HTTP 错误摘要和鉴权提示。

商业验证必须在项目后端环境完成：

- 后端 GB28181 网关能成功 invite 设备。
- invite 返回的 FLV/TS/HLS/WebRTC/fMP4 URL 能在浏览器播放。
- BYE/PTZ/query/keepalive 被后端接受并产生预期设备行为。

没有后端网关和真实设备/模拟设备证据时，不把 GB28181 标记为商业可用。
